# OneNoteToNotion 项目原则

## 核心价值观

1. **数据完整性优先**: 同步过程必须确保数据不丢失、不损坏，宁可降级提示也不能静默丢失内容

2. **用户体验可预测**: 同步结果应当透明，用户能清楚知道哪些内容成功、哪些失败、哪些被降级处理

3. **最小侵入性**: 不修改原始 OneNote 数据，只进行读取和转换

## 技术原则

### 架构设计
- **分层解耦**: 解析(Parsing) → 映射(Mapping) → 同步(Sync) 三层分离
- **接口抽象**: 依赖抽象接口而非具体实现，便于测试和扩展
- **不可变数据**: 数据传输对象使用不可变设计，避免副作用

### 代码质量
- **明确错误处理**: 每层都捕获并包装异常，提供有意义的错误信息
- **诊断友好**: 支持输出诊断信息，便于问题排查
- **类型安全**: 利用 C# 类型系统减少运行时错误

### 性能考虑
- **COM 线程亲和**: OneNote COM 组件必须在 STA 线程上顺序调用
- **API 并发写入**: Notion API 调用可以并发执行，提升同步速度
- **内存控制**: 大文件（如图片）需控制内存占用

## 决策框架

### 功能实现优先级
1. **数据不丢失** > 功能完整性 > 性能优化
2. **降级处理** > 抛出异常中断同步
3. **用户可感知** > 静默处理

### 技术选型标准
- 优先选择与现有架构一致的方案
- Windows 平台优先使用系统原生能力
- 避免引入重量级依赖

### 错误处理策略
- 可恢复错误：记录日志，继续处理
- 不可恢复错误：标记失败项，提供重试机制
- 格式不支持：降级为占位符，提示用户

## 同步质量标准

### 支持的内容类型
- 标题 (Heading 1-3)
- 段落 (Paragraph)
- 列表 (Bulleted/Numbered)
- 表格 (Table)
- 图片 (Image) - 待实现
- 附件 (Attachment) - 待实现

### 降级处理规则
- 复杂样式：简化为基本样式 + 降级提示
- 超大表格：分块处理
- 超大图片：自动缩图
- 不支持元素：占位符 + 说明

## 安全与隐私
- Notion Token 通过配置传递，不硬编码
- 不上传任何数据到第三方服务
- 本地处理，直接同步到用户指定的 Notion workspace
